# Pupbox 硬件路线图

第一个硬件里程碑，是做出一台由家长陪同使用、纯语音交互的原型。孩子是否愿意反复使用语音能力还没有充分验证前，暂不增加动作、屏幕和自动唤醒。

## 调研结论

- FoloToy 已经验证过火火兔改造方案：使用 ESP32、火火兔自带的模拟麦克风，并把语音处理放在服务器端；设备通过 MQTT 接收控制消息、上传录音，再通过 HTTPS 下载回复音频。其公开的短回复目标是 3–5 秒。
- FoloToy 提供适配火火兔的固件和可自行部署的服务端，但它使用的 MQTT、UDP 和 HTTPS 设备协议不能直接连接 Pupbox，需要增加协议适配层。
- 小智 AI 给出的常见 DIY 组合包括：ESP32-S3 N16R8、INMP441 麦克风、MAX98357A 功放和 2–3W 扬声器。成本较低，但需要自己接线、处理供电并适配固件。
- 集成式 ESP32-S3 音频开发板已经把主控、麦克风、音频编解码或功放、扬声器接口整合在一起，更适合第一次做硬件，能减少接线和音频干扰问题。

## 推荐实施顺序

### H0：完成手机端体验验收

进入硬件阶段前，软件应满足：

- 孩子可以连续玩 10 分钟，不会因为延迟、上下文断裂或机械重复而放弃。
- 家长可以通过诊断页理解每一轮失败的原因，不需要读取原始录音。
- 在家长直接陪同下，安全回复和最大音量可以接受。

### H1：不装外壳的桌面硬件原型

购买一块集成式 ESP32-S3 音频开发板，先使用一个实体按键、板载麦克风和小扬声器。原型阶段使用 USB 连接合格的移动电源供电，不要立即接裸锂电池。

设备固件第一版只需要完成：

1. 连接 Wi-Fi，并保存设备访问令牌。
2. 按住按键时录音。
3. 把录音上传到 Pupbox 设备接口。
4. 下载并播放服务端返回的 Opus 音频。
5. 播放“正在听、正在等待、正在说话、发生错误”等简单提示音。

STT、会话状态、安全规则、Qwen、TTS 和诊断能力继续放在现有 Go 服务里，ESP32 不运行大模型。

### H2：装进毛绒小狗

选择带可拆卸电子仓的毛绒外壳，麦克风孔和扬声器开口不能被厚填充物挡住。录音按键应做成凹陷式或用布料覆盖，并确保无法被孩子拆下。

第一版毛绒原型继续使用 USB 供电，或者使用有完整外壳和保护电路的成品移动电源。需要测试：表面温度、最大音量、拉线强度、跌落后是否松动，以及是否存在可吞咽的小零件。

### H3：免按键和动作能力

只有 H2 稳定后再增加：

- 本地 VAD 和唤醒词。
- 回声处理，避免小狗听到自己的扬声器后误触发。
- 一个低风险、速度较慢的动作，例如尾巴摆动。
- 带保护电路的电池、充电板、固定外壳和可维护结构。

## 购买成品还是自己开发

### 最快的实体体验验证

购买 FoloToy Magicbox，或者型号明确、与 FoloToy 固件兼容的魔改火火兔。这样可以最快验证“实体玩具是否比手机更能吸引孩子”，但它不能直接复用 Pupbox，需要协议适配，还可能引入单独的订阅和固件依赖。

### 推荐的 Pupbox 原型路线

购买集成式 ESP32-S3 音频开发板，并继续使用现有 Pupbox 后端。它需要开发一小段设备固件，但可以完整保留我们已经完成的儿童对话策略、诊断、安全规则、模型选择和部署能力。

### 不建议作为第一步的路线

不要一开始就购买裸 ESP32-S3、独立麦克风、独立功放、充电板和裸电池，再自行塞入毛绒玩具。FoloToy 的公开记录显示，仅充电干扰和麦克风选型就可能消耗大量排查时间。第一次硬件迭代使用集成音频开发板更稳妥。

## 淘宝咨询参数清单

可以把下面这段直接发给店家：

> 我需要做一个按键对话的 AI 语音玩具。请确认开发板是否使用 ESP32-S3，至少 8MB Flash 和 8MB PSRAM；是否板载麦克风、音频 Codec、D 类功放和喇叭接口；是否支持 2.4GHz Wi-Fi、USB 烧录、I2S 全双工录音播放；是否提供 ESP-IDF 或 Arduino 的录音、HTTPS 下载和音频播放源码。请说明麦克风型号、Codec 型号、功放功率、适配的喇叭阻抗、板子尺寸、电池接口型号、充电管理和低电压保护情况。最好能提供 Ogg Opus 或 MP3 网络音频播放例程，以及充电时麦克风底噪测试结果。

建议参数如下：

| 项目 | 最低要求 | 推荐配置 |
| --- | --- | --- |
| 主控 | ESP32-S3 | ESP32-S3，双核 240MHz |
| Flash | 8MB | 16MB |
| PSRAM | 8MB | 8MB 或 16MB |
| 网络 | 2.4GHz Wi-Fi | 板载天线，最好可选外置天线 |
| 麦克风 | 1 个板载 MEMS 或模拟麦克风 | 双麦阵列，有可运行的降噪/AEC示例 |
| 录音 | 16kHz、16bit、单声道 | 支持 16/24kHz PCM/WAV |
| 音频输出 | I2S Codec 或 DAC + 功放 | ES8311 或同等级 Codec + D 类功放 |
| 扬声器 | 明确支持一种喇叭规格 | 4Ω 2–3W 或 8Ω 1–3W 腔体喇叭 |
| 按键 | 至少一个可编程按键或可用 GPIO | 录音键、停止键、音量键均可扩展 |
| 开发 | USB 烧录、原理图、引脚表 | ESP-IDF 示例完整，Arduino 可选 |
| 音频格式 | 能播放 WAV/PCM | 有 HTTP Ogg Opus 或 MP3 解码示例 |
| 电源 | USB-C 5V | 电池充电、电源路径和电量检测齐全 |

以下能力不是第一版必需：屏幕、摄像头、SD 卡、IMU、蓝牙、RGB 灯阵列、离线大模型。不要为了这些功能牺牲音频质量、板子尺寸或开发资料。

## H1 简单架构

```text
按住说话按键
    |
    v
板载麦克风 -> I2S/Codec -> ESP32-S3 的 PSRAM 录音缓冲
                                  |
                                  | HTTPS 上传 WAV
                                  v
                            日本 VPS / Pupbox
                    STT -> 会话状态 -> Qwen -> TTS
                                  |
                                  | HTTPS 下载 Ogg Opus
                                  v
ESP32-S3 软件解码 -> I2S/Codec -> D 类功放 -> 1~3W 扬声器
```

设备还需要四类简单反馈：

- 正在录音：一个短提示音或常亮 LED。
- 正在等待：低音量、间隔较长的提示音。
- 正在播放：停止录音，避免收到自己的扬声器声音。
- 网络或服务错误：一个固定错误音，然后允许再次按键。

第一版供电建议：

```text
USB-C 5V -> 开发板电源管理 -> ESP32-S3 / Codec / 功放
```

桌面原型先接合格的成品移动电源。进入毛绒外壳后，再使用带保护板的 3.7V 锂电池，并确认开发板具备充电管理、电源路径、过充过放保护或搭配受保护电芯。物理总开关应当能真正断开主电源。

## 是否使用树莓派

当前不把树莓派作为最终路线。树莓派运行 Linux，开发固件更容易，也可以直接运行 Go 客户端，但体积、功耗、启动时间、发热、SD 卡可靠性和电池设计都不适合第一只毛绒玩具。

树莓派只保留为备用调试方案：如果 ESP32 音频驱动或 Opus 解码长时间无法打通，可以用树莓派 Zero 2 W 加 USB 音频设备快速验证硬件形态；验证完成后仍回到 ESP32-S3。由于所有大模型计算都在 VPS，Pupbox 不需要树莓派提供额外算力。
